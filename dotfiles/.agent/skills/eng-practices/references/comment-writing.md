# レビューコメント作法と Pushback ハンドリング

Google eng-practices の Reviewer Guide のうち「How to write code review comments」と「Handling pushback」を、Claude Code エージェントがコメントを書くとき・著者と議論するときに直接使える形に整理する。

## レビューコメントの書き方

### 5 つの基本

1. **コードに焦点を当てる**。「あなたは…」ではなく「このコードは…」。人格批判にしない。
2. **理由を添える**。「ここはこう変えてほしい」だけでは弱い。スタイルガイド、測定値、過去のバグ、可読性の根拠を 1 行で添える。
3. **重要度を明示する**。マージ可否に直結する指摘と、好みレベルの提案を混ぜない。
4. **良い実装を褒める**。修正案を要求するだけでなく、注目すべき改善点や賢い実装を肯定する。レビューは指導の場でもある。
5. **コメントよりコード**: 「読みにくいのでコメントを足してほしい」よりも「読みやすくなるよう関数を分けてほしい」を優先する。コード自体を直せるなら直す。

### 重要度ラベル

| ラベル | 意味 | マージ可否 |
|---|---|---|
| `Critical:` または無印 | マージ前に直すべき correctness / security / data loss / production regression | 修正必須 |
| `Warning:` | 修正または明確な説明が必要なリスク | 修正または合意必須 |
| `Nit:` | 些末な好み・スタイルの提案 | 任意（マージブロックしない） |
| `Optional:` | 改善案だが採否は著者に委ねる | 任意 |
| `FYI:` | 情報共有・関連事項 | 対応不要 |

`pr-code-review` の severity（Critical / Warning / Suggestion）と整合させる：
- `Critical:` → Critical
- `Warning:` → Warning
- `Nit:` / `Optional:` / `FYI:` → Suggestion

### コメントテンプレート

```
[severity ラベル] [問題の短い記述]。
理由: [スタイルガイド / 測定値 / 過去バグ / 仕様]。
提案: [具体的な修正案、できればコードスニペット]。
```

例:

```
Warning: この refresh は cache の lock を取らないので、複数 goroutine から呼ばれると重複 fetch する。
理由: 過去 incident #1234 と同じ race。
提案: sync.Mutex で fetch 全体を guard、または singleflight に置き換える。
```

### コメントの粒度

- 同じ問題が複数箇所にあるなら 1 箇所だけ詳細にコメントし、他は「同様」と短くする。
- 大量に Nit を残すと著者が Critical を見落とす。Nit はまとめて 1 コメントに集約するか、しきい値以下なら省略する。
- 大きな設計問題が見えたら、他の細部コメントを後回しにして設計問題だけ先に出す（[reviewer-guide.md](reviewer-guide.md) §3 を参照）。

## Pushback ハンドリング（レビュアー側）

著者が「変更したくない」と返してきたとき。

1. **著者の主張を評価する**。著者はコードに最も近い人物で、レビュアーが知らない文脈を持っていることが多い。
2. **正しければ取り下げる**。技術的に説得力があれば、こだわらない。レビュアーの役目は code health の確保であって自尊心の確保ではない。
3. **code health のために主張を続けるなら追加情報を出す**。「私の好み」ではなく「過去のバグ／スタイルガイド／測定値」を根拠にする。
4. **礼儀と事実を両立する**。攻撃的にならない。著者を萎縮させると次回以降のレビューも難しくなる。
5. **「後で直す」を信用しない**。実績ベースで「後で」はほぼ来ない。新 CL/issue を切ってリンクを残すよう要求する。
6. **エスカレーション**: 数往復で合意しないなら、チーム/技術リーダーに上げる。沈黙して放置しない。

## Pushback への対応（著者側）

[cl-author-guide.md](cl-author-guide.md) の「Handling reviewer comments」を参照。基本姿勢は:

- 個人化しない。
- まずコード自体の改善を試す（読みにくさは説明より構造で直す）。
- 事実ベースで合意形成する。
- 採用しない場合は理由を明示し、必要なら issue/TODO を切る。

## 建設的でないコメントへの対処

- 攻撃的・人格批判的なコメントが来た場合は、公開コメント欄で反論しない。
- 私的なチャットや 1on1 で対話し、改善がなければチーム/マネージャーへエスカレーションする。
- Claude Code エージェントが指摘する立場のときも、攻撃的・断定的なトーンを避け、根拠のない強い言葉は使わない。
