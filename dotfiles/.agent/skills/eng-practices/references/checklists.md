# 役割別チェックリスト

短文・プロンプト的に使うことを想定した、4 場面のチェックリスト。SKILL.md 本体や reviewer-guide / comment-writing / cl-author-guide の要点を当てはめる順序で並べてある。

## 実装者向け（コードを書いて PR を出す直前）

- [ ] CL の目的は 1 文で言えるか（混在していないか）。
- [ ] テストを同 CL に入れたか。難しい場合は理由と代替検証を CL 説明に書いたか。
- [ ] 行数は 100 行前後／400 行以内か。超える場合、再分割を検討したか。
- [ ] 公開 API の互換性影響を CL 説明に書いたか（破壊的変更があるなら明示）。
- [ ] コードコメントは Why を書いているか。What の重複コメントは消したか。
- [ ] 命名は説明的か。新規読者が短時間で理解できる構造になっているか。
- [ ] build / lint / test / 型検査の結果を集めたか。未実行があれば理由をメモ。
- [ ] PR タイトルは命令形・具体的・50〜70 文字程度か。
- [ ] PR 本文に Why / What / 影響 / 代替案 / 残課題 / 参照 が揃っているか。

## レビュアー向け（レビュー開始から完了まで）

- [ ] CL 説明を最初に読んだか。設計が概念的に妥当か判断したか。
- [ ] 主ロジックのファイルから読んでいるか。
- [ ] テストを早めに読んで意図を確認したか。
- [ ] 大きな設計問題が見えたら、細部コメントより先に出したか。
- [ ] 各コメントに重要度ラベル（`Critical:` / `Warning:` / `Nit:` / `Optional:` / `FYI:`）を付けたか。
- [ ] 理由（スタイルガイド・過去バグ・測定値）を 1 行で添えたか。
- [ ] 良い実装に肯定的フィードバックを残したか。
- [ ] 「読みにくい」は説明追加ではなくコード改善で直せないかを優先したか。
- [ ] 応答は 1 営業日以内に返したか。集中で遅らせた場合、半日以内に区切ったか。
- [ ] 1000 行超など過大 CL は分割依頼を最優先に出したか。

## PR 説明を書く人向け

- [ ] タイトルは命令形・短く・具体的か。
- [ ] Why を本文に書いたか（背景・動機・捨てた代替案）。
- [ ] What を要約として書いたか（実装の主要ポイント）。
- [ ] 公開 API / データ / 運用への影響を明示したか。
- [ ] ロールバック計画があるなら本文に書いたか。
- [ ] 未検証項目を未検証として明示したか（嘘の検証完了報告をしていないか）。
- [ ] 関連 issue / 設計ドキュメント / 過去 CL / 再現手順をリンクしたか。
- [ ] 悪い例（`Bug fix` / `Update` / `WIP` だけ）になっていないか。

## レビュー指摘を受ける人向け

- [ ] 個人化していないか（感情で反応していないか）。
- [ ] まず「コード自体で直せないか」を検討したか（説明追加より構造改善）。
- [ ] 反論するときは事実ベース（スタイルガイド・測定値・過去 incident）で根拠を示したか。
- [ ] 採用しないコメントには理由と代替対応（issue/TODO リンク）を返したか。
- [ ] 「後で直す」を口頭約束で済ませず、issue/TODO を切ったか。
- [ ] 全コメント対応後、再レビュー依頼を明示したか。
- [ ] force push 必須 repo なら、変更点を PR コメントで要約したか。
- [ ] 攻撃的なコメントには公開で反論せず、私的対話・エスカレーションに回したか。
