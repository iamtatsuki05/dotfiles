# エラーパターン別の調査メモ

一般的なエラーの意味の解説はここには書かない。見落としやすい gotcha と、エラー種別を問わない横断的な調査手順・判断基準だけをまとめる。

## 横断的な調査手順

1. スタックトレースは最下部（最も深い呼び出し）から読み、失敗行の変数の型・値を確認する
2. `git diff` / `git log -p` で直近の変更が失敗行・その呼び出し元に関係していないか確認する
3. 問題の変数は代入箇所を rg で洗い出し、None/null/想定外の値が入る経路（条件分岐・エラーパス）を特定する
4. 「動く環境と動かない環境がある」場合は、コードより先に環境差分を比較する: 言語バージョン、環境変数、設定ファイル、lock ファイルと実際にインストールされたものの差

## 見落としやすい gotcha

- **Import / パス**: パスの大文字小文字の不一致は、macOS（大文字小文字非区別 FS）では動き Linux/CI で初めて壊れる
- **ネットワーク**: dev/staging/prod の URL が環境変数で切り替わっていない混在が頻出。401/403 はコードより先にトークン期限切れ・未設定を疑う
- **環境**: `.env.example` だけあって `.env` が未コピー、pyenv/nodenv/goenv 等の version manager の設定漏れ
- **テスト失敗**: 単体では通るのに suite 全体で落ちる場合は、テスト間の状態汚染（実行順序依存）を疑う。モックが本物の挙動と乖離していないかも確認する
- **DB**: `no such table` / カラム不明はまずマイグレーション未実行を疑う。制約違反は挿入データを実際にダンプして確認する
- **並行処理**: Go は読解より先に `go test -race ./...` / `go run -race` で data race を検出する。デッドロックはロック取得順序の非対称を探す

## 判断基準

- null チェック追加か上流修正か: null がドメイン上正当な状態なら呼び出し側でチェックし、バグ由来なら上流で発生させない修正を優先する
- 原因候補が複数残る場合は、最も安く反証できる仮説から print デバッグで潰す（SKILL.md Phase 2 の調査ループ）
